# Отчет по аудиту безопасности (Задание 7)

Проект: форма на WSGI + MariaDB.

## 1) XSS

### Риск
Вывод пользовательских данных в HTML без экранирования приводит к выполнению внедренного JavaScript.

### Защита
- Для HTML-атрибутов и строк используется `html.escape(...)`.
- В рендере формы и админ-страниц все пользовательские поля экранируются перед вставкой.

### Пример из кода
- `wsgi.py`: функции `_h(...)`, `_render_main(...)`, `_render_admin_page(...)`, `_render_admin_edit_page(...)`.

## 2) Information Disclosure

### Риск
Показ пользователю внутренних исключений/трейсов раскрывает структуру приложения и окружение.

### Защита
- Пользователю возвращаются нейтральные сообщения об ошибках.
- Подробности исключений выводятся только при `WSGI_DEBUG=1/true/yes`.

### Пример из кода
- `wsgi.py`: общий `except Exception as exc` в `application(...)`.
- `wsgi.py`: обработка ошибки подключения к БД с условным добавлением деталей.

## 3) SQL Injection

### Риск
Формирование SQL-строк с пользовательским вводом позволяет изменить запрос.

### Защита
- Для всех операций с пользовательскими данными используются параметризованные запросы (`%s` + tuple/list параметров).
- CRUD в пользовательской и админской части (insert/update/delete/select) выполняется только через prepared statements.

### Пример из кода
- `wsgi.py`: `_insert_submission(...)`, `_update_submission(...)`, `_upsert_submission_languages(...)`.
- `wsgi.py`: admin-операции `/admin/delete`, `/admin/edit`, login-проверка `/login`.

## 4) CSRF

### Риск
Сторонний сайт может заставить браузер жертвы отправить POST-запрос от ее имени.

### Защита
- Для гостя: double-submit cookie (`guest_csrf`) + скрытое поле `csrf_token`.
- Для авторизованного пользователя: серверный `csrf_token` хранится в `user_sessions`, сверяется в каждом POST.
- Для админки: отдельный `admin_csrf` cookie + hidden field для edit/delete форм.

### Пример из кода
- `wsgi.py`: `_csrf_valid(...)`.
- `wsgi.py`: проверки CSRF в `POST /`, `POST /login`, `POST /logout`, `POST /admin/delete`, `POST /admin/edit`.

## 5) Include

### Риск
Подстановка пользовательского пути в include/file-read может дать доступ к произвольным файлам (LFI/RFI).

### Защита
- Раздача статики ограничена каталогом `/static`.
- Запрещены переходы по `..` и абсолютные пути.
- Динамического include по пользовательским данным в приложении нет.

### Пример из кода
- `wsgi.py`: `_serve_static(...)` проверяет `..`, абсолютные пути и существование файла в `static`.

## 6) Upload

### Риск
Небезопасная загрузка файлов может привести к RCE, XSS, переполнению хранилища.

### Состояние в проекте
- Функциональности загрузки файлов в приложении нет.
- Поверхность атаки Upload отсутствует.

### Рекомендации при добавлении Upload в будущем
- Белый список MIME/расширений.
- Ограничение размера файлов.
- Хранение вне webroot.
- Генерация случайных имен файлов.
- Проверка содержимого и отключение исполнения.

## Вывод

Для текущего набора функций реализованы базовые защиты от XSS, SQL Injection, CSRF, утечки информации, а также исключены Include/Upload-риски в существующей архитектуре.
